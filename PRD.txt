OVERVIEW
--------
A Flutter Android app for Indian index option trading via Dhan, backed by a
thin FastAPI service running on a Linux VM.

Flutter owns all heavy computation:
  - Indicator calculation (7 functions ported from Python to Dart)
  - Signal engine logic
  - Dhan REST + WebSocket calls (candles, OI, orders)
  - Telegram Bot API calls (tips)
  - Live trade state machine

FastAPI backend is intentionally thin — no WebSockets, no computation:
  - User registration + JWT auth
  - Trade history sync (POST after exit, GET for history)
  - Local push notification trigger (optional v1)
  - Health check endpoint

No Flask-style WebSocket complexity in the backend at all.
All real-time data flows Flutter <-> Dhan directly.

Target: Android 8.0+ (API 26+), single-platform Flutter build.
Backend: Python 3.11+, FastAPI, PostgreSQL 16, deployed on existing Linux VM.

          +-- GET  /trades/stats      today / weekly / all-time P&L summary
          +-- GET  /health            liveness check


--------------------------------------------------------------------------------
ARCHITECTURE PRINCIPLES
--------------------------------------------------------------------------------

1. FLUTTER OWNS ALL REAL-TIME COMPUTATION
   - Candle fetching, EMA, regime, phase, health, score, signal -- all in Dart.
   - No server round-trip on the hot path (5s dashboard refresh).
   - Dashboard works fully offline (last cached candles).

2. FASTAPI IS A THIN DATA LAYER ONLY
   - No WebSockets on the backend.
   - No indicator computation on the backend.
   - No Dhan API calls on the backend.
   - Only persists and serves structured records (users, trades).
   - ~300 lines of Python total for v1.

3. DUAL PERSISTENCE
   - SQLite (local): candles, instrument master, tips cache, trade draft.
   - PostgreSQL (server): confirmed completed trades, user accounts.
   - On trade exit: Flutter writes SQLite first, then syncs to FastAPI.
   - If offline: local SQLite is source of truth; sync when reconnected.

4. DIRECT DHAN REST API FROM FLUTTER
   - Base URL: https://api.dhan.co
   - Auth headers set once in dio interceptor:
       client-id: <CLIENT_ID>
       access-token: <ACCESS_TOKEN>
   - Each user stores their own Dhan credentials (encrypted, AES-256).

5. TELEGRAM VIA BOT API (no MTProto complexity)
   - User creates a bot, adds it to channel as admin.
   - App polls GET /getUpdates every 5 min (tips screen open).
   - Bot token stored encrypted in local SQLite.

6. JWT AUTH FLOW
   - access_token:  1-hour expiry, stored in flutter_secure_storage.
   - refresh_token: 30-day expiry, stored in flutter_secure_storage.
   - dio interceptor auto-refreshes access_token on 401.
   - Logout: DELETE refresh token from server + clear secure storage.

7. LOCAL NOTIFICATIONS (no FCM for v1)
   - flutter_local_notifications handles SL hit, target hit, new BUY signal.
   - Foreground service keeps trade monitoring alive when app is backgrounded.
   - FCM (server-push) deferred to v2 (requires Firebase project setup).

MODULE 1 -- USER ACCOUNTS & AUTHENTICATION
--------------------------------------------------------------------------------

DESCRIPTION
  Hybrid auth: local PIN for quick device access + server JWT for trade sync.
  On first launch user registers on FastAPI -> gets JWT -> stored securely.
  PIN is a device-level shortcut; it does not replace the server account.

SCREENS
  1.1  Splash / Account Picker Screen (app launch)
       - If no accounts registered: show "Get Started" -> Register Screen.
       - If accounts exist: show list of saved profiles (name + avatar initial).
       - "Add Account" button at bottom.
       - Tap account -> PIN Entry Screen.

  1.2  Register Screen
       - Fields:
           Name         (text, 2-40 chars)
           Email        (text, validated format)
           Password     (text, obscured, min 8 chars)
           Confirm Pwd  (text, must match)
           PIN          (4-digit numpad, confirmed twice)
       - "Register" button -> POST /auth/register -> save JWT in secure_storage.
       - On success: navigate to Dhan Credential Setup.

  1.3  Login Screen (returning user, new device)
       - Email + Password fields.
       - "Login" -> POST /auth/login -> save tokens -> PIN setup for this device.

  1.4  PIN Entry Screen
       - 4-digit numpad overlay (full-screen modal).
       - 3 wrong attempts -> lock for 5 min (stored in SharedPreferences).
       - On success: decrypt Dhan access_token from SQLite -> set in dio.
       - Biometric unlock option (local_auth package) -- optional, toggle in settings.

  1.5  Account Settings Screen
       - Change PIN.
       - Change Password (requires current password -> PUT /auth/password).
       - Delete account (with confirmation -> DELETE /auth/account).
       - Link / update Dhan credentials.
       - Link Telegram.
       - Logout (clears tokens + redirects to Login).

MODULE 2 -- DHAN ACCOUNT LINKING & TRADE EXECUTION
--------------------------------------------------------------------------------

DESCRIPTION
  Each user links exactly one Dhan account. Credentials are encrypted in SQLite.
  Trade flow: idle -> watching -> ordering -> active -> exiting -> idle (same as web).

2A. DHAN CREDENTIAL SETUP
  Screen: Link Dhan Account
  - Fields: Client ID (text), Access Token (text, long, obscured).
  - "Test Connection" button:
      -> GET /fundlimit with provided creds
      -> On success: shows "Balance: Rs.XX,XXX" in green.
      -> On failure: shows error code from Dhan API.
  - On save: UPSERT into dhan_credentials (AES encrypt access_token), status='ok'.
  - Re-verification: tap "Re-verify" any time (e.g., after token expiry).

  Dhan REST: GET https://api.dhan.co/fundlimit
  Headers: client-id: <CLIENT_ID>   access-token: <ACCESS_TOKEN>
  Response: { "availabelBalance": float, "withdrawableBalance": float }

2B. CUSTOM TRADE SETUP SCREEN
  Mirrors web app /custom form exactly.

  Fields:
    Instrument    (dropdown): NIFTY | BANKNIFTY | FINNIFTY | MIDCPNIFTY | SENSEX
    Strike        (number input, integer)
    Option Type   (toggle CE / PE)
    Entry Price   (decimal): buy trigger -- enter when LTP crosses above this value
    Stop Loss     (decimal): must be < entry
    Targets       (up to 3, comma-separated): e.g. "180, 220"
    Lots Mode     (toggle Auto / Manual)
    Lots          (integer, visible only when Manual mode)

  Validation:
    - SL must be < entry price.
    - Target(s) must be > entry price.
    - If auto lots: entry price required first (used for lot calc).
    - Strike must exist in instrument_master for today's expiry.

  On "Confirm" tap:
    Step 1 -- Instrument lookup:
      - Query local SQLite instrument_master for matching row:
          WHERE instrument_name = 'OPTIDX'
            AND trading_symbol LIKE '%<NIFTY_SYMBOL>%'
            AND ABS(strike_price - <strike>) < 1
            AND option_type = '<CE|PE>'
            AND expiry_date >= date('now')
          ORDER BY expiry_date ASC LIMIT 1
      - If no match or last_updated < today: re-download CSV, re-query.
      - Result fields needed: security_id, trading_symbol, lot_size, expiry_date.

    Step 2 -- Auto lot calculation (if auto mode):
      - GET /fundlimit -> availableBalance
      - lots = floor((balance / 2) / (entry x lot_size))
      - lots = max(1, min(lots, 20))    -- cap at 20 lots

    Step 3 -- Navigate to Live Trade Screen with params:
      { instrument, strike, optionType, tradingSymbol, securityId,
        lotSize, lots, quantity (= lots x lotSize),
        entry, sl, targets: [t1, t2?, t3?] }

  Instrument Master CSV:
    URL: https://images.dhan.co/api-data/api-scrip-master.csv
    Download: once per trading day (check last_updated date in SQLite).
    Relevant columns:
      SEM_TRADING_SYMBOL    -> trading_symbol
      SEM_SMST_SECURITY_ID  -> security_id
      SEM_STRIKE_PRICE      -> strike_price
      SEM_OPTION_TYPE       -> option_type  ('CE' or 'PE')
      SEM_INSTRUMENT_NAME   -> instrument_name (filter = 'OPTIDX')
      SEM_EXPIRY_DATE       -> expiry_date  (format: "DD-MMM-YYYY")
      SEM_LOT_UNITS         -> lot_size
      SEM_EXM_EXCH_ID       -> exchange     ('NSE' or 'BSE')
    Parse with Dart `csv` package. Store ~50k rows in SQLite.
    Index: CREATE INDEX idx_master ON instrument_master(instrument_name, option_type, expiry_date);

  LOCAL SQLITE SCHEMA -- instrument_master table:
    CREATE TABLE instrument_master (
      id               INTEGER PRIMARY KEY AUTOINCREMENT,
      trading_symbol   TEXT,
      security_id      TEXT,
      strike_price     REAL,
      option_type      TEXT,
      instrument_name  TEXT,
      expiry_date      TEXT,
      lot_size         INTEGER,
      exchange         TEXT,
      last_updated     TEXT
    );
    CREATE INDEX idx_master ON instrument_master(instrument_name, option_type, expiry_date);

2C. LIVE TRADE SCREEN
  Mirrors web app /live page in full detail.

  Header:
    Trading symbol (e.g., NIFTY2540722900PE)
    Expiry date + CE/PE badge + lots x lot_size = total qty
    State banner: WATCHING | ORDERING | ACTIVE | EXITING (colour-coded)

  Price section:
    Live LTP (large, colour green/red vs entry)
    Entry trigger  <- from setup
    Buy zone upper = (entry + target[0]) / 2
    Progress bar: entry -> target (fills as LTP moves toward target)

  Trade params:
    Buy Price   (shown once ACTIVE)
    Stop Loss   (updates when trailed -- highlighted when changed)
    Target 1    (strikethrough when hit)
    Target 2    (if set, dims after T1 hit)
    Order ID    (shown once ordered)
    P&L         (live, large, green/red -- shown once ACTIVE)

  Buttons:
    WATCHING: "Cancel" (back to custom screen)
    ACTIVE:   "Manual Exit" (red, triggers SELL MARKET)

  WebSocket Feed (Dhan MarketFeed):
    URL: wss://api-feed.dhan.co
    Connection lifecycle:
      - Connect on screen open (WATCHING state)
      - Disconnect on trade exit or screen pop
    Auth packet (send immediately after connect):
      {
        "LoginReq": {
          "MsgCode": 11,
          "ClientId": "<CLIENT_ID>",
          "Token": "<ACCESS_TOKEN>"
        }
      }
    Subscribe packet (send after auth ACK):
      {
        "RequestCode": 21,
        "InstrumentCount": 1,
        "InstrumentList": [
          {
            "ExchangeSegment": "NSE_FNO",
            "SecurityId": "<SECURITY_ID>"
          }
        ]
      }
    Incoming tick (binary, parse with ByteData):
      Offset  0: uint8  feed_response_code (= 2 for Quote)
      Offset  1: uint8  message_length
      Offset  2-9: uint8[8] exchange_segment + security_id
      Offset 10-13: float32 LTP
      Offset 14-17: float32 LTT (epoch)
      Offset 18-21: float32 LTQ
      Offset 22-25: float32 avg_trade_price
      Offset 26-29: float32 volume
      Offset 30-33: float32 total_sell_qty
      Offset 34-37: float32 total_buy_qty
      Offset 38-41: float32 open
      Offset 42-45: float32 close (prev day)
      Offset 46-49: float32 high
      Offset 50-53: float32 low
      Offset 54-57: float32 change_percent
    Reconnect: exponential backoff 1s -> 2s -> 4s -> 8s -> 30s (max).
    Re-subscribe after reconnect.

  State Machine (identical to Python web app logic):

    WATCHING:
      On each tick:
        if ltp > entry AND ltp < buyZoneUpper:
          -> place BUY MARKET order -> state = ORDERING

    ORDERING:
      Wait for order status confirmation:
        Poll GET /orders/<orderId> every 2s until status = 'TRADED'
        On TRADED: set buyPrice = ltp (or fill price from order),
                   state = ACTIVE
        On REJECTED/CANCELLED: state = WATCHING (retry allowed)
        Timeout 30s: show error banner, stay WATCHING

    ACTIVE:
      On each tick:
        Trailing SL logic:
          bar = target1 - buyPrice
          if ltp >= buyPrice + (bar / 2) AND NOT trailed:
            newSL = round(buyPrice + (bar / 4), 0.5)
            update sl = newSL, trailed = true
            show "SL Trailed to <newSL>" local notification

        SL check:  if ltp <= sl -> trigger exit (reason = 'SL')
        T1 check:  if ltp >= targets[0]:
                     if targets.length == 1: trigger exit (reason = 'TARGET')
                     else: sell 50% qty -> order, move SL = midpoint,
                           t1Hit = true, show T1 notification
        T2 check:  if t1Hit AND ltp >= targets[1]:
                     trigger exit remaining qty (reason = 'TARGET2')

      Manual exit: user taps "Manual Exit" -> trigger exit (reason = 'MANUAL')

    EXITING:
      Place SELL MARKET order for remaining qty.
      Poll order status until TRADED.
      On TRADED:
        exitPrice = fill price
        pnl = (exitPrice - buyPrice) x quantity
        Save to local SQLite trades table.
        POST to FastAPI /trades (sync).
        Show local notification: "Trade closed -- P&L: Rs.<pnl>"
        Navigate to Trade Summary Screen.
      On failure: banner "Exit failed -- retrying", stay ACTIVE.

  Dhan REST -- Place Order:
    POST https://api.dhan.co/orders
    Body:
    {
      "dhanClientId":      "<CLIENT_ID>",
      "transactionType":   "BUY" | "SELL",
      "exchangeSegment":   "NSE_FNO",
      "productType":       "INTRADAY",
      "orderType":         "MARKET",
      "validity":          "DAY",
      "securityId":        "<SECURITY_ID>",
      "quantity":          <int>,
      "price":             0,
      "triggerPrice":      0,
      "disclosedQuantity": 0,
      "afterMarketOrder":  false
    }
    Response: { "orderId": str, "orderStatus": str }

  Dhan REST -- Order Status:
    GET https://api.dhan.co/orders/<orderId>
    Response: {
      "orderId": str,
      "orderStatus": "PENDING" | "TRADED" | "REJECTED" | "CANCELLED",
      "tradedPrice": float,
      "tradedQuantity": int
    }

2D. TRADE SUMMARY SCREEN
  Shown after exit completes.
  Fields:
    Symbol + expiry + direction (CE/PE)
    Buy Price | Exit Price | P&L (large, green/red)
    Exit reason: SL HIT | TARGET 1 | TARGET 2 | MANUAL EXIT

2E. TRADE HISTORY SCREEN

  Filters:
    Date range picker

  Export:
    "Export CSV" button -> generates CSV, shares via Android share sheet.
    CSV columns: date, time, symbol, direction, buy_price, exit_price,

  Sync indicator:

      Retry button -> sync now.

MODULE 3 -- TELEGRAM TIPS
--------------------------------------------------------------------------------

DESCRIPTION
  App reads trading tips from a Telegram channel via Bot API.
  User creates a bot (@BotFather), adds it as admin to the channel.
  App polls for new messages every 5 min (when tips screen is visible).

3A. TELEGRAM SETUP SCREEN
  Fields:
    Bot Token   (text, obscured) -- from @BotFather
    Channel ID  (text) -- "@channelname" or numeric "-1001234567890"
  Buttons:
    "Test" -> calls GET /getMe (validate token) + GET /getUpdates (show count)
    "Save" -> encrypt bot_token with AES-256, store in telegram_sessions

  Test flow:
    1. GET https://api.telegram.org/bot<TOKEN>/getMe
       -> verify response ok=true, show bot username
    2. GET .../getUpdates?limit=1
       -> show "Last message: <date>" or "No messages yet"

3B. TIPS LIST SCREEN
  Header: channel name + "Last synced: <time>" chip
  Pull-to-refresh -> fetch latest messages.
  Auto-refresh: every 5 min while screen is visible (Timer.periodic).

  Fetch flow:
    GET .../getUpdates?offset=<last_update_id+1>&limit=50&allowed_updates=["channel_post"]
    For each update:
      Extract message text + date + message_id.
      Run is_tip() filter.
      If is_tip -> run parse_tip() -> upsert into tips_cache.
    Update last_update_id in SQLite telegram_sessions.

  is_tip(text):
    - text must start with '#' (after stripping whitespace)
    - text (lowercase) must contain at least one of:
        'sl', 'stoploss', 'stop loss', 'target', 'pe', 'ce', 'call', 'put'

  parse_tip(text):
    symbol:    regex #([A-Z]+)  (first hashtag word, uppercased)
    strike:    regex \b(\d{4,6})\b  (first 4-6 digit number)
    opt_type:  regex \b(PE|CE)\b  (case-insensitive, first match)
    entry:     regex (?:above|entry|buy\s*above)\s*([\d.]+)  (first match)
    sl:        regex (?:sl|stoploss|stop\s*loss)\s*[:\-]?\s*([\d.]+)  (first)
    targets:   regex (?:target|tgt)\s*[:\-]?\s*([\d./]+)
               split result on '/' -> list of strings
    raw:       full message text
    date:      epoch -> IST formatted "DD MMM YYYY HH:MM"
    msg_id:    Telegram message_id (for deduplication)

  Tip Card UI (card per tip, newest first):
    Row 1: #NIFTY badge (coloured) | 22900 | CE/PE badge | date-time
    Row 2: Entry: 160  .  SL: 145  .  T1: 180  .  T2: 210
    Row 3 (truncated, expandable): raw message text (2 lines, tap to expand)
    Bottom-right: "Execute" button (filled, primary colour)

  Execute flow:
    1. Tap "Execute" on tip card.
    2. Navigate to Custom Trade Setup Screen pre-filled:
         instrument = tip.symbol
         strike     = tip.strike
         option_type= tip.opt_type
         entry      = tip.entry
         sl         = tip.sl
         targets    = tip.targets.join(", ")
    3. User reviews params, adjusts lots.
    4. Confirms -> Live Trade Screen.

  LOCAL SQLITE SCHEMA -- tips_cache table:
    CREATE TABLE tips_cache (
      id         INTEGER PRIMARY KEY AUTOINCREMENT,
      user_id    INTEGER REFERENCES users(id),
      msg_id     INTEGER NOT NULL,
      symbol     TEXT,
      strike     TEXT,
      opt_type   TEXT,
      entry      TEXT,
      sl         TEXT,
      targets    TEXT,        -- JSON array e.g. '["180","220"]'
      raw        TEXT,
      date_str   TEXT,
      fetched_at TEXT,
      UNIQUE(user_id, msg_id)
    );
    CREATE INDEX idx_tips ON tips_cache(user_id, fetched_at DESC);

MODULE 4 -- STRATEGY DASHBOARD
--------------------------------------------------------------------------------

DESCRIPTION
  Real-time strategy dashboard mirroring the web app strategy page.
  All computation in Dart. Candles from Dhan API, cached in SQLite.
  5s refresh cycle; OI refresh every 30s.

4A. DATA PIPELINE

  Candle fetch schedule:
    On screen open:
      - Load last 50 candles from SQLite for selected instrument.
      - If < 6 candles: trigger immediate API fetch.
    Every 5-min boundary + 35s:
      - Fetch completed 5-min candles for today.
      - UPSERT into SQLite.
    Every 5s (live partial bar):
      - Fetch 1-min candles from current 5-min boundary to now.
      - Aggregate into partial OHLCV bar (do not persist).
      - Use for latest close/spot price.

  Candle API:
    GET https://api.dhan.co/charts/intraday
    Params:
      securityId       = <index_id>  (NIFTY=13, BANKNIFTY=25, FINNIFTY=27,
                                      MIDCPNIFTY=442, SENSEX=51)
      exchangeSegment  = IDX_I
      instrument       = INDEX
      interval         = 5 (or 1 for live partial)
      fromDate         = YYYY-MM-DD (today)
      toDate           = YYYY-MM-DD (today)
    Response:
      { "timestamp": [epoch, ...],  <- NOTE: "timestamp" not "start_Time"
        "open": [...], "high": [...], "low": [...],
        "close": [...], "volume": [...] }
    Epoch conversion: epoch_seconds -> DateTime.fromMillisecondsSinceEpoch(
                        epoch * 1000, isUtc: true).toLocal()
                      -> format "YYYY-MM-DD HH:MM:SS" (IST UTC+5:30)

  OI fetch schedule (when OI tracking enabled):
    Step 1 -- Get expiry:
      GET https://api.dhan.co/optionchain/expirylist
      Params: underlyingSecurity=<id>, underlyingType=INDEX
      Response: { "data": { "data": ["DD-MM-YYYY", ...] } }
      -> pick soonest date >= today

    Step 2 -- Get option chain:
      POST https://api.dhan.co/v2/optionchain
      Body: {
        "UnderlyingScrip":   <int>,
        "UnderlyingSegment": "IDX_I",
        "Expiry":            "YYYY-MM-DD"
      }
      Response: {
        "data": {
          "last_price": float,
          "oc": {
            "<STRIKE>": {
              "ce": { "oi": int, "volume": int, "iv": float, "ltp": float, ... },
              "pe": { "oi": int, "volume": int, "iv": float, "ltp": float, ... }
            },
            ...
          }
        }
      }
    Parse: ATM = strike nearest to last_price.
    Extract ATM +/- 5 strikes.
    For each strike: ce_oi, pe_oi, ce_ltp, pe_ltp, ce_iv, pe_iv.
    Compute: total_ce_oi, total_pe_oi, pcr = total_pe_oi / total_ce_oi.
    Track prev snapshot for delta (ce_delta, pe_delta).

  LOCAL SQLITE SCHEMA -- candles table:
    CREATE TABLE candles (
      id          INTEGER PRIMARY KEY AUTOINCREMENT,
      instrument  TEXT NOT NULL,
      interval_m  INTEGER NOT NULL,
      time        TEXT NOT NULL,
      open        REAL,
      high        REAL,
      low         REAL,
      close       REAL,
      volume      INTEGER,
      UNIQUE(instrument, interval_m, time)
    );
    CREATE INDEX idx_candles ON candles(instrument, interval_m, time DESC);

  LOCAL SQLITE SCHEMA -- oi_snapshots table (optional, for OI delta calc):
    CREATE TABLE oi_snapshots (
      id          INTEGER PRIMARY KEY AUTOINCREMENT,
      instrument  TEXT NOT NULL,
      snapshot_at TEXT NOT NULL,
      strike      INTEGER,
      ce_oi       INTEGER,
      pe_oi       INTEGER,

4B. INDICATOR COMPUTATION (port from indicators_dashboard.py)

  File: lib/indicators/indicators.dart
  All functions are pure (no I/O, no side effects).

  Input data model:
    class Candle {
      final String time;
      final double open, high, low, close;
      final int volume;
    }

  ---------------------------------------------------------
  1. List<double?> computeEma(List<Candle> candles, {int period = 9})
  ---------------------------------------------------------
    k = 2.0 / (period + 1)
    result = List<double?>(candles.length)
    ema = null
    for i in 0..candles.length-1:
      close = candles[i].close
      ema = ema == null ? close : close * k + ema * (1 - k)
      result[i] = i >= period - 1 ? round(ema, 2) : null
    return result

  ---------------------------------------------------------
  2. String classifyRegime(List<Candle> candles12, List<double?> ema12)
  ---------------------------------------------------------
    Returns: "IMPULSE_UP" | "IMPULSE_DOWN" | "REVERSAL_WATCH" | "CONSOLIDATION"
    Logic:
      emaSlope = (ema12.last ?? 0) - (ema12[ema12.length-4] ?? ema12.last ?? 0)
      aboveCount = count of candles12 where close > ema (non-null ema)
      belowCount = valid - aboveCount
      crossover: last close < ema AND prev close > ema (bearish cross)
      crossunder: last close > ema AND prev close < ema (bullish cross)
      if crossover || crossunder: return "REVERSAL_WATCH"
      if aboveCount >= 8 && emaSlope > 0: return "IMPULSE_UP"
      if belowCount >= 8 && emaSlope < 0: return "IMPULSE_DOWN"
      return "CONSOLIDATION"

  ---------------------------------------------------------
  3. Map<String,dynamic> computeMoveVelocity(List<Candle> candles6)
  ---------------------------------------------------------
    Returns: { "velocity": double, "type": "SHARP"|"GRIND"|"FLAT" }
    netMove = abs(candles6.last.close - candles6.first.open)
    velocity = round(netMove / candles6.length, 2)
    type = velocity > 15 ? "SHARP" : velocity >= 5 ? "GRIND" : "FLAT"

  ---------------------------------------------------------
  4. String classifyMovePhase(List<Candle> candles, List<double?> ema,
                              List<Map> oiHistory, String regime)
  ---------------------------------------------------------
    Returns: "BASE"|"BREAKOUT"|"TREND_RIDE"|"EXHAUSTION"|"REVERSAL"
    Waterfall logic:
      // Reversal check
      if regime == "REVERSAL_WATCH": return "REVERSAL"
      // Base check (tight range)
      range = max(high) - min(low) over last 6 candles
      avgRange = average(high-low) per candle over last 12
      if range < avgRange * 1.5 AND velocity.type == "FLAT": return "BASE"
      // Breakout check
      rangeExpansion = last candle range > avgRange * 1.8
      oiBuildup = oiHistory.isNotEmpty AND
                  (oiHistory.last.total_ce_delta + oiHistory.last.total_pe_delta) > 0
      if rangeExpansion || oiBuildup: return "BREAKOUT"
      // Exhaustion check
      last3 = candles.last 3
      avgBody = average body/range ratio over last 3 candles
      if avgBody < 0.3 AND velocity < 5: return "EXHAUSTION"
      // Default
      return "TREND_RIDE"

  ---------------------------------------------------------
  5. Map<String,dynamic> computeTrendHealth(List<Candle> candles12,
       List<double?> ema12, double callDelta, double putDelta,
       List<double> pcrSeries)
  ---------------------------------------------------------
    Returns: { "score": int, "warnings": List<String> }
    Start: score = 100, warnings = []
    Deductions:
      Volume drop: last vol < avg(prev 5 vol) x 0.7 -> -10, warn "Volume declining"
      Call OI surge: callDelta > putDelta x 1.5 -> -15, warn "Heavy call writing (bearish pressure)"
      PCR falling: pcrSeries.length >= 3 AND last 3 vals decreasing -> -15, warn "PCR declining"
      Close below EMA: candles12.last.close < ema12.last -> -20, warn "Price below EMA9"
      Velocity drop: velocity of last 3 < velocity of prev 3 -> -10, warn "Momentum dropping"
      Doji: last candle body/range < 0.15 -> -10, warn "Doji candle -- indecision"
    score = max(0, score)
    return { "score": score, "warnings": warnings }

  ---------------------------------------------------------
  6. Map<String,dynamic> computeLinearMoveScore(
       String regime, Map velocity, double pcrNow, double ivPct,
       double oiDirection, double volRatio, double structScore)
  ---------------------------------------------------------
    Returns: { "score": int, "signal": "ENTER"|"WAIT"|"AVOID", "breakdown": Map }
    Weights: emaSlopeW=0.20, oiBuildupW=0.25, pcrTrendW=0.15, ivDirW=0.15,
             candleStructW=0.15, volumeRatioW=0.10
    Component scores (0-100 each):
      emaSlope:      IMPULSE_UP/DOWN -> 90, REVERSAL_WATCH -> 20, else 50
      oiBuildup:     oiDirection mapped linearly (-100..+100 -> 0..100)
      pcrTrend:      pcrNow > 1.2 -> 80, > 1.0 -> 60, > 0.8 -> 40, else 20
      ivDirection:   ivPct (already 0-100, from iv_skew_ratio)
      candleStruct:  structScore (already 0-100)
      volumeRatio:   min(volRatio / 2.0, 1.0) x 100
    weightedScore = sum(component x weight for each)
    signal = score >= 65 ? "ENTER" : score >= 40 ? "WAIT" : "AVOID"
    breakdown = { "ema_slope": s1, "oi_buildup": s2, "pcr_trend": s3,
                  "iv_direction": s4, "candle_structure": s5, "volume_ratio": s6 }

  ---------------------------------------------------------
  7. Map<String,dynamic> detectOiWall(Map<int,Map> strikeOiMap, double spot)
  ---------------------------------------------------------
    Returns: { resistanceStrike, resistanceOi, supportStrike, supportOi,
               nearestWallDistance }
    resistanceStrikes = strikes above spot, sorted ascending
    supportStrikes    = strikes below spot, sorted descending
    Find max CE OI among resistanceStrikes -> resistanceStrike
    Find max PE OI among supportStrikes    -> supportStrike
    nearestWall = min(abs(resistanceStrike - spot), abs(supportStrike - spot))

  Additional helpers (lib/indicators/helpers.dart):
    double computeIvPercentile(Map? oiSnap):
      ratio = oiSnap["iv_skew_ratio"] ?? 1.0
      return clamp((ratio - 0.9) / 0.2 * 100, 0, 100)

    double computeOiDirection(List<Map> oiRows):
      net = sum(r["pe_delta"] - r["ce_delta"])
      total = sum(abs(r["pe_delta"]) + abs(r["ce_delta"]))
      if total == 0: return 0.0
      return clamp(net / total * 100, -100, 100)

    double computeVolumeRatio(List<Candle> candles):
      vols = candles.map(c.volume).where(v > 0).toList()
      if vols.length < 2: return 1.0
      lastVol = vols.last
      prior = vols.sublist(max(0, vols.length-21), vols.length-1)
      priorAvg = prior.isEmpty ? 1.0 : prior.reduce(+) / prior.length
      return priorAvg == 0 ? 1.0 : round(lastVol / priorAvg, 3)

    double computeCandleStructure(List<Candle> last4):
      scores = last4.map((c) {
        r = c.high - c.low
        b = abs(c.close - c.open)
        return r > 0 ? b / r : 0.0
      })
      return round(scores.reduce(+) / scores.length * 100, 1)

4C. SIGNAL ENGINE (port from signal_engine.py)

  File: lib/indicators/signal_engine.dart

  Constants (mirror Python values exactly):
    buyPhaseTriggers  = {"BREAKOUT", "TREND_RIDE"}
    exitPhaseTriggers = {"EXHAUSTION", "REVERSAL"}
    minLinearScore    = 65
    minHealthScore    = 50
    noTradeHealth     = 40
    noTradeLinear     = 40
    cooldownMinutes   = 15
    targetMult        = 1.125     // entry x 1.125 = target
    slMult            = 0.9375    // entry x 0.9375 = SL

  Per-instrument state (in-memory, reset on app restart):
    Map<String, Map> _signalState = {}
    Keys: action, direction, time (DateTime)

  Map<String,dynamic> generateSignal({
    required String instrument,
    required String regime,
    required String phase,
    required Map<String,dynamic> health,
    required Map<String,dynamic> linearScore,
    required Map<String,dynamic> velocity,
    Map<String,dynamic>? oiSnap,
    double? spot,
  })

  Step 1 -- Collect counter reasons:
    List<String> counter = []
    if phase in exitPhaseTriggers:
      counter.add("Phase is $phase -- momentum fading, consider exit")
    if regime == "REVERSAL_WATCH":
      counter.add("EMA crossover detected -- reversal in progress")
    if regime == "CONSOLIDATION" && phase != "BREAKOUT":
      counter.add("Market in consolidation -- no directional edge")
    if healthScore < noTradeHealth:
      counter.add("Trend health weak ($healthScore/100)")
    if linScore < noTradeLinear:
      counter.add("Linear score low ($linScore/100)")
    if velType == "FLAT" && regime in {"IMPULSE_UP","IMPULSE_DOWN"}:
      counter.add("Velocity flat despite impulse regime")
    if counter.isNotEmpty -> return _dedup(NO_TRADE signal)

  Step 2 -- Buy conditions:
    if regime in {"IMPULSE_UP","IMPULSE_DOWN"}
       && phase in buyPhaseTriggers
       && linScore >= minLinearScore
       && healthScore >= minHealthScore
       && velType != "FLAT":
         direction = regime == "IMPULSE_UP" ? "CE" : "PE"
         (entry, atmStrike) = _getEntry(direction, oiSnap)
         if entry != null:
           target = _roundPremium(entry * targetMult)
           sl     = _roundPremium(entry * slMult)
           return _dedup(BUY signal)
         else:
           return _dedup(WAIT -- buy conditions met but OI tracker not running)

  Step 3 -- WAIT:
    reason = _waitReason(regime, phase, linScore, healthScore, velType)
    return _dedup(WAIT signal)

  double _roundPremium(double v):
    // Rounds to nearest 0.5 (option tick)
    return (v * 2).round() / 2.0

  Map _dedup(String instrument, Map sig):
    prev = _signalState[instrument] ?? {}
    stateChanged = sig.action != prev.action || sig.direction != prev.direction
    cooldownOk = true
    if !stateChanged && sig.action == "BUY":
      prevTime = prev.time
      if prevTime != null:
        elapsed = DateTime.now().difference(prevTime).inMinutes
        cooldownOk = elapsed >= cooldownMinutes
    if stateChanged || (sig.action == "BUY" && cooldownOk):
      sig.isNew = true
      _signalState[instrument] = { action, direction, time: DateTime.now() }
    return sig

FASTAPI DEPLOYMENT ON LINUX VM
--------------------------------------------------------------------------------

DIRECTORY LAYOUT ON VM

  /opt/marketmascot/
    +-- webapp/          existing Flask app (already running)
    +-- backend/         new FastAPI backend

STEP 1 -- PostgreSQL setup

  sudo apt install postgresql postgresql-contrib -y
  sudo systemctl enable --now postgresql

  sudo -u postgres psql <<EOF
  CREATE USER mm_user WITH PASSWORD 'STRONG_PASSWORD_HERE';
  CREATE DATABASE marketmascot OWNER mm_user;
  GRANT ALL PRIVILEGES ON DATABASE marketmascot TO mm_user;
  EOF

STEP 2 -- Clone / copy backend files

  sudo mkdir -p /opt/marketmascot/backend
  sudo chown $USER:$USER /opt/marketmascot/backend
  # Copy backend/ directory to /opt/marketmascot/backend/

STEP 3 -- Python virtualenv for backend

  cd /opt/marketmascot/backend
  python3.11 -m venv .venv
  source .venv/bin/activate
  pip install -r requirements.txt

STEP 4 -- Environment file

  cat > /opt/marketmascot/backend/.env <<EOF
  DATABASE_URL=postgresql+asyncpg://mm_user:STRONG_PASSWORD_HERE@localhost:5432/marketmascot
  SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")
  ACCESS_TOKEN_EXPIRE_MINUTES=60
  REFRESH_TOKEN_EXPIRE_DAYS=30
  ALLOWED_ORIGINS=*
  APP_ENV=production
  EOF
  chmod 600 /opt/marketmascot/backend/.env

STEP 5 -- Database migration (create tables)

  cd /opt/marketmascot/backend
  source .venv/bin/activate
  alembic upgrade head

STEP 6 -- Systemd service for FastAPI

  sudo tee /etc/systemd/system/marketmascot-api.service <<EOF
  [Unit]
  Description=MarketMascot FastAPI Backend
  After=network.target postgresql.service

  [Service]
  Type=exec
  User=www-data
  Group=www-data
  WorkingDirectory=/opt/marketmascot/backend
  EnvironmentFile=/opt/marketmascot/backend/.env
  ExecStart=/opt/marketmascot/backend/.venv/bin/uvicorn main:app \
            --host 127.0.0.1 --port 8000 --workers 2 \
            --log-level info --access-log
  Restart=on-failure
  RestartSec=5s
  StandardOutput=journal
  StandardError=journal
  SyslogIdentifier=mm-api

  [Install]
  WantedBy=multi-user.target
  EOF

  sudo systemctl daemon-reload
  sudo systemctl enable marketmascot-api
  sudo systemctl start marketmascot-api
  sudo systemctl status marketmascot-api

STEP 7 -- Systemd service for Flask webapp (if not already set up)

  sudo tee /etc/systemd/system/marketmascot-webapp.service <<EOF
  [Unit]
  Description=MarketMascot Flask Web App
  After=network.target

  [Service]
  Type=exec
  User=www-data
  WorkingDirectory=/opt/marketmascot/webapp
  EnvironmentFile=/opt/marketmascot/webapp/.env
  ExecStart=/opt/marketmascot/webapp/.venv/bin/gunicorn \
            -k geventwebsocket.gunicorn.workers.GeventWebSocketWorker \
            -w 1 --bind 127.0.0.1:5000 app:app
  Restart=on-failure
  RestartSec=5s

  [Install]
  WantedBy=multi-user.target
  EOF

  sudo systemctl daemon-reload
  sudo systemctl enable marketmascot-webapp
  sudo systemctl start marketmascot-webapp

STEP 8 -- Nginx configuration

  sudo apt install nginx -y

  sudo tee /etc/nginx/sites-available/marketmascot <<'EOF'
  # Flask web app
  server {
      listen 80;
      server_name app.yourdomain.com;

      location / {
          proxy_pass         http://127.0.0.1:5000;
          proxy_http_version 1.1;
          proxy_set_header   Upgrade $http_upgrade;
          proxy_set_header   Connection "upgrade";
          proxy_set_header   Host $host;
          proxy_set_header   X-Real-IP $remote_addr;
          proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
          proxy_read_timeout 86400;     # keep alive for WebSocket
      }
  }

  # FastAPI backend
  server {
      listen 80;
      server_name api.yourdomain.com;

      location / {
          proxy_pass         http://127.0.0.1:8000;
          proxy_set_header   Host $host;
          proxy_set_header   X-Real-IP $remote_addr;
          proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
          proxy_read_timeout 60;
      }

      location /health {
          proxy_pass http://127.0.0.1:8000/health;
          access_log off;
      }
  }
  EOF

  sudo ln -s /etc/nginx/sites-available/marketmascot \
             /etc/nginx/sites-enabled/
  sudo nginx -t
  sudo systemctl reload nginx

--------------------------------------------------------------------------------
SECURITY CONSIDERATIONS
--------------------------------------------------------------------------------

  1. Dhan access_token encrypted at rest (AES-256-CBC).
     Encryption key = SHA-256(user_PIN + user_email).
     Decrypted in memory only during API calls, never logged.

  2. Bot token similarly encrypted in SQLite.

  3. PIN stored as SHA-256 hash, never plaintext.

  4. JWT access_token: 1h expiry, stored in flutter_secure_storage (hardware-
     backed keystore on Android 6+, not SharedPreferences).
     Refresh_token: 30d expiry, also in secure_storage.

  5. On 3 failed PIN attempts: account locked 5 min
     (counter in SharedPreferences, time-checked on each attempt).

  6. No network calls made until PIN authentication succeeds.

  7. Android manifest:
     android:allowBackup="false"    prevents adb backup of SQLite.
     android:usesCleartextTraffic="false"  enforces HTTPS.

  8. FastAPI backend:
     bcrypt (cost 12) for password hashing -- no MD5/SHA for passwords.
     Refresh tokens stored as SHA-256 hash in DB, not raw token.
     CORS locked to app domain in production (not "*").
     Soft-delete users (is_active=false) to preserve trade history.

  9. PostgreSQL:
     mm_user has no superuser privileges.
     Credentials in .env with chmod 600.
     DB not exposed externally (localhost only, Nginx is the gateway).

  10. SSL/TLS enforced end-to-end (Certbot HTTPS on api subdomain).
